# Asset Management System — Итоговая архитектура

**Статус:** DRAFT, согласовано с пользователем 2026-04-21. Черновик концепции, не план реализации.

## Концепция UI
- **Навигация:** постоянный Sidebar слева.
- **Главный паттерн:** Side-panel (Drawer) — клик по элементу (активу/сотруднику/филиалу) выдвигает панель справа с деталями. Контекст не теряется, скорость работы максимальна.
- **Таблицы — только где уместно.** Филиалы — НЕ таблица, а Grid красивых карточек.
- **Super Admin принцип (zero-hardcode):** категории, отделы, статусы, поставщики — всё редактируется из Settings, хранится в `settings/*`, не живёт в коде.

## Страницы

### 1. Панель (`/dashboard`, super_admin)
- Виджеты-счётчики (вещи / в ремонте / лицензии активные / сотрудники)
- Секция "Внимание": гарантии/лицензии истекают через ≤ 30 дней
- Лента аудита: последние 5-10 действий

### 2. Склад (`/warehouse`, все)
- Фильтр по умолчанию: `holderType='storage'` (только то что реально на складе)
- Таблица + inline-кнопка **[Выдать]** на строке
- Клик [Выдать] → Drawer c поиском по сотрудникам/отделам/филиалам → одно нажатие подтверждает
- Клик по строке → Drawer с деталью актива (фото, все поля, история)
- Переключатель режима "только склад / все активы" для полного обзора когда надо

### 3. Сотрудники (`/employees`, admin+)
- Таблица: Имя · Отдел · Филиал · Кол-во вещей · Кол-во лицензий
- Клик → Drawer: список закреплённых активов + софта
- Действие "Уволить" → генерирует чеклист возврата вещей, отмечает как `archived`

### 4. Филиалы (`/branches`, admin+)
- **НЕ таблица.** Grid карточек: название, адрес, итог по мебели и технике
- Клик по карточке → страница филиала (split-view): слева список вещей этого филиала, справа Drawer с деталью выбранной

### 5. Лицензии (`/licenses`, все)
- Верх: карточки-индикаторы ("Adobe 8/10", "Office 365 50/100")
- Таблица: видно к чему привязана лицензия — `deviceId` или `employeeId`
- Клик → Drawer с деталью

### 6. Настройки (`/settings`, super_admin)
- **Конструктор категорий:** создание новых типов (Электросамокаты, Автомобили и пр.)
- **Справочники CRUD:** Категории · Отделы · Филиалы · Сотрудники · Лицензии · Статусы ремонта · Поставщики
- Все изменения идут в `settings/global_lists/*` и сразу подхватываются всеми выпадающими списками приложения

## Решения (2026-04-21, от пользователя)

1. **Филиалы ≠ Сотрудники.** Вещь прикреплена либо к адресу филиала, либо к сотруднику — НЕ одновременно. Это два независимых measurement'а. `/branches/:id` показывает только те активы, что прикреплены к адресу филиала (`holderType='storage'` + `branchId`, или новый `holderType='branch'`). `/employees/:id` показывает активы `holderType='user'` + `holder.userId = :id`. Пересечения нет.

2. **"Уволить" = блок, не удаление.** Пользователь переходит в `status='blocked'` (или эквивалент). Документ НЕ удаляется — сохраняется вся история и логи для аудита. Блок снимает доступ в систему.

3. **Возврат активов при увольнении — чеклист с зелёными галочками, которые ставит Админ.** Можно отмечать как по одной вещи, так и группой. Визуал: список из 10 вещей → 9 зелёных + 1 красная (не вернули/потеряно). Состояние чеклиста сохраняется (не теряется между сессиями). Пока не закрыт чеклист — пользователь в `status='blocking'` или `pending_return`, после закрытия → `blocked`.

4. **Главный роут `/`** — остаётся HomeTiles (плитки-быстрые-ссылки на разделы) как удобный launcher. Пользователь явно сказал "давай действовать как планировали" — значит плитки не трогаем.

5. **Мобильный Drawer** — default full-screen modal (проще, в MVP хватит). Bottom-sheet — отложено.

## Доменные правки (перед реализацией)

- `Asset.holderType`: добавить вариант `'branch'` ИЛИ использовать `'storage' + branchId` — выбор implementer-а, главное документировать.
- `User.status`: ввести значения `active | blocking | blocked | archived`. Не удалять поле `archived`, расширить enum.
- Новая коллекция или подколлекция: **ReturnChecklist** (per user-offboarding), связанная с `users/{uid}/returns/{returnId}` или верхнеуровневая `returns/{returnId}`. Поля: `userId`, `createdBy`, `createdAt`, `status: open|closed`, `items: [{ assetId, returned: bool, note, returnedAt, confirmedBy }]`, `closedAt`. Правила: создать/редактировать — admin+, читать — admin+ и сам userId.
- Все изменения согласовать с firestore.rules ПЕРЕД имплементацией.
